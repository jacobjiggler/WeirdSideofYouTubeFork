# Project Rules

## Before declaring any task done

1. Take a screenshot of https://weirdtube.wtf using the browser/playwright tools and visually confirm the page looks correct — not just a 200 status code.
2. For any UI change (CSS, templates, layout, new pages), also screenshot the specific page or element that changed and confirm it renders as intended.
3. If the site is down or looks broken, investigate and fix before wrapping up — never declare done on a broken site.
4. For CSS/JS changes, verify against a fresh fetch (`fetch(url, {cache: 'no-store'})`) or a brand-new browser session, not just a re-navigate in an already-open Playwright session. Both Cloudflare's edge cache (~4h) and the browser's own HTTP cache can silently keep serving the pre-change bundle to a session that already loaded the page earlier, making a real fix look broken (or a real regression look fine).
5. Layout rules that depend on "space available below the navbar" (or similar) must not use a fixed vh/vw guess — the navbar's height changes (e.g. logged out vs. logged-in admin with more menu items). Compute it from actual layout (`getBoundingClientRect`/`innerHeight`) instead.
6. A unit test on the sizing math (e.g. `computePortraitHeight`) only proves the arithmetic is self-consistent — it does NOT prove the real page doesn't scroll, because it never renders real CSS/layout (margins, other elements on the page, box-sizing, etc. are all invisible to it). It is not a substitute for step 7, only a supplement.
7. For the homepage player specifically (and any similar "fits in one screen" layout), after any change to it, always check for a vertical scrollbar on the main page after clicking "Get Weird" — `document.documentElement.scrollHeight <= window.innerHeight` should hold. Check this at more than one viewport height (e.g. ~860px and a shorter ~700px) and in both the logged-out nav and the logged-in admin nav (which is taller), since the previous bugs here only showed up in the taller-nav/shorter-viewport combination. Do this live against weirdtube.wtf per rule 4, not from a stale cached session.
8. When manually or automatically (e.g. Playwright) testing live against weirdtube.wtf, start the session with `?notrack=1` on the first request (e.g. `https://weirdtube.wtf/?notrack=1`) — this opts that whole session out of the engagement-stats analytics (see `lib/analyticsGate.ts`), so verification traffic doesn't pollute the real avg-videos-watched/bounce-rate/session-ending-video numbers on `/admin/stats`. The flag persists for the rest of that session once set, so it only needs to be on the very first request.
